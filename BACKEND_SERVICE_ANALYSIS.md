# Backend Service Architecture Analysis & Refactoring Roadmap

This document analyzes the current complexity of the `apps/websocket` service layer and proposes a strategic path toward a more maintainable and scalable architecture.

## 1. Current Issues (The "Complexity" Root Causes)

### A. The "God Service" (GameFlowService)

`GameFlowService` currently acts as a central hub for everything. It handles:

- **Application Logic**: Managing socket inputs and validations.
- **Domain Orchestration**: Deciding when a game should start or when a turn ends.
- **Infrastructure**: Directly calling `publisher.emit...` to send WebSocket events.
- **State Management**: Manually clearing and starting timeouts (`ReadyTimeoutManager`, `TurnTimeoutManager`).

### B. Logic Fragmentation

Business rules are scattered. For example, the check for "Is the game ready to start?" exists in both `GameFlowService` and `RoomService`. This makes it easy for the state to get out of sync.

### C. Procedural Flow & Mutation

Services often reach deep into objects to mutate them (e.g., `RoomRegistry` setting `player.isReady = status`). This bypasses encapsulation and makes it difficult to track state changes.

### D. Service Proxying

`RoomService` often acts as a simple pass-through to `RoomRegistry` or `GameSessionManager`, adding an unnecessary layer of indirection without adding clear value.

## 2. Proposed Architectural Shift

### Phase 1: Separation of Concerns (Domain vs. Infrastructure)

- **Domain Layer**: Keep pure game logic in `@ttt/core`. The backend should treat `@ttt/core` as the source of truth for game state.
- **Application Layer**: Introduce **Use Cases** or **Command Handlers**. Instead of one large `GameFlowService`, create small, focused classes:
  - `HandlePlayerReadyUseCase`
  - `ProcessPlayerMoveUseCase`
  - `HandleTimeoutUseCase`
- **Infrastructure Layer**: Treat WebSockets as a "Side Effect". Services should not know about `Socket` objects or `Publisher` methods.

### Phase 2: Domain-Driven Events

Instead of a service calling `publisher.emitNextTurn()`, the Domain objects (e.g., `Room` or `Game`) should emit events:

1. `GameAction` occurs -> Domain updates state -> Domain emits `NextTurnEvent`.
2. An **Event Dispatcher** catches `NextTurnEvent`.
3. A **Socket Subscriber** receives the event and calls `publisher.emitNextTurn()`.

### Phase 3: Room State Machine

Implement a formal State Machine for `Room` transitions:

- `LOBBY` -> (Player Joins) -> `WAITING_FOR_READY`
- `WAITING_FOR_READY` -> (All Ready) -> `PLAYING`
- `PLAYING` -> (Winner/Draw) -> `GAME_OVER`
- `PLAYING` -> (Timeout) -> `GAME_OVER`

### Phase 4: Simplified Repository Pattern

Replace `RoomRegistry` and `GameSessionManager` with a more cohesive **Repository Pattern**:

- `RoomRepository`: Handles storage and retrieval of `Room` objects.
- `UserRepository`: Handles user data.

## 3. Concrete Example: Refactoring `handleReady`

**Current (Procedural):**

1. Get roomId/userId from socket.
2. Call `roomService.readyPlayer`.
3. If successful, manually call `publisher.emitPlayerReady`.
4. Manually check if everyone is ready.
5. If yes, manually call `roomService.gameStart`.
6. Manually reset ready states.
7. Manually call `publisher.emitPlaying`.

**Proposed (Command-Based):**

1. Socket receives `READY` event.
2. Socket dispatches `SetReadyCommand(userId, roomId, status)`.
3. `ReadyCommandHandler` handles it:
   - Fetches `Room` from `RoomRepository`.
   - Calls `room.toggleReady(userId, status)`.
   - `Room` object updates its internal state and emits a `RoomUpdatedEvent`.
4. `SocketEventListener` hears `RoomUpdatedEvent` and broadcasts the new state to the room.
5. If the `Room` state transitions to `PLAYING`, it emits a `GameStartedEvent`, which triggers the game start logic independently.

## 4. Immediate Next Steps (Low-Hanging Fruit)

1. **Move Validation Logic**: Move basic input validation (e.g., move index 0-8) into a DTO or a validation utility.
2. **Decouple Publisher**: Pass an event listener to services instead of the hard-coded `GameEventPublisher`.
3. **Encapsulate State**: Stop direct mutations in `RoomRegistry`; add methods to the `Room` model to handle its own state changes.
