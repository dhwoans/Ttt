# 📝 개인 프로젝트 Git 관리 원칙 (Git Usage Principles for Solo Projects)

이 원칙들은 프로젝트의 **버전 이력을 명확히** 하고, **안정적인 코드베이스를 유지**하며, 필요할 때 **과거로 쉽게 돌아갈 수 있게** 하기 위해 존재합니다.

---

## 1. 🌲 버전 관리 전략: 핵심 브랜치 유지

복잡한 브랜치 전략 대신, **두 가지 핵심 브랜치**만 사용하여 효율성을 높입니다.

| 브랜치 종류 | 역할 및 원칙 | 명명 규칙 (예시) |
| :--- | :--- | :--- |
| **`main`** | 1. **[최종 안정 버전]** 실제 배포 가능한 상태의 코드만 유지합니다. | `main` |
| | 2. `main` 브랜치에서는 직접 작업하지 않고, **새 브랜치에서 작업 후 병합(Merge)**만 합니다. | |
| **작업 브랜치** | 1. 새로운 기능 개발, 리팩토링, 버그 수정 등 **모든 개발 작업**을 분리하여 진행합니다. | `feature/login-ui` |
| | 2. 작업 완료 및 테스트 후, `main`으로 병합하고 **반드시 삭제**하여 이력을 깔끔하게 유지합니다. | `fix/issue-404` |
| | 3. 브랜치 이름에는 **작업의 목적**을 명확히 드러냅니다. | `refactor/typescript-migration` |

---

## 2. ✏️ 명확한 커밋 기록 (Commit Message Clarity)

**미래의 내가 이 코드를 왜 바꿨는지** 빠르게 이해할 수 있도록 커밋 메시지를 명확히 작성하는 것이 핵심입니다.

### 2-1. 커밋 주기 및 구조

* **자주 커밋:** 하나의 논리적인 작업 단위가 끝날 때마다 커밋하여, 되돌릴 수 있는 작은 단위를 확보합니다.
* **구조:** `<유형>: <제목 (50자 이내, 명령형)>` 형식으로 작성합니다.

### 2-2. 커밋 유형 (Type) 접두사 사용

커밋 제목의 시작에 다음 중 하나의 접두사를 사용하여 **변경의 목적**을 나타냅니다.

| 유형 | 목적 | 예시 |
| :--- | :--- | :--- |
| `feat` | **새로운 기능** 추가 | `feat: 사용자 프로필 수정 페이지 구현` |
| `fix` | **버그** 수정 | `fix: 서버 연결 끊김 시 재시도 로직 수정` |
| `refactor` | 코드 구조 **개선** (기능 변화 없음) | `refactor: 유틸 함수 분리 및 리네이밍` |
| `docs` | **문서** 변경 (README, 주석 등) | `docs: 기능 구현 상세 주석 추가` |
| `chore` | 빌드, 환경 설정, 라이브러리 업데이트 등 **비(非)코드 변경** | `chore: ESLint 규칙 추가 및 적용` |

> 💡 **커밋 메시지 예시:** `feat: 메인 페이지에 사용자 이름 표시 기능 추가`

## 3. 🛠️ 이력 정리 및 안전성 확보 (History Management & Safety)

### 3-1. 커밋 이력 수정 및 정리 (로컬에만 있을 때)

로컬 저장소에만 존재하는 커밋은 아래 명령어를 통해 규칙에 맞게 정리할 수 있습니다.

| 명령어 | 목적 | 사용 상황 | 비고 |
| :--- | :--- | :--- | :--- |
| **`git commit --amend`** | **가장 최근 커밋 수정** | 직전 커밋 메시지가 잘못되었거나, 빠진 파일을 추가하고 싶을 때. | 새 커밋 ID가 생성됩니다. |
| **`git rebase -i HEAD~n`** | **여러 커밋을 재정리** | 여러 개의 작은 커밋을 하나의 깔끔한 커밋으로 **합치고(Squash)** 싶을 때, 또는 과거 커밋 메시지를 수정할 때. | `n`은 수정할 커밋의 개수입니다. |
| **`git revert <commit-id>`** | **안전하게 되돌리기** | 특정 커밋에서 했던 변경 내용을 취소하는 **새로운 커밋**을 만듭니다. | 이력이 안전하게 보존됩니다. |
| **`git reset --hard`** | **로컬 작업 강제 초기화** | 커밋 이력과 **로컬 작업 내용까지 모두 삭제**하고 특정 시점으로 돌아갈 때. | **데이터 손실 위험**이 있으므로 매우 신중하게 사용해야 합니다. |

### 3-2. 원격 저장소는 보험이다

* **자주 푸시 (Push):** 작업 완료 시점에 **항상 원격 저장소 (GitHub 등)에 코드를 푸시**하여 코드 유실을 막습니다.

### 3-3. 임시 저장에는 `stash`

* 현재 작업 중인 내용을 커밋하기 애매한 상태에서 다른 급한 작업을 해야 할 경우, **`git stash`**를 사용하여 변경 사항을 임시로 보관합니다.
