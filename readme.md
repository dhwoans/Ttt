100개의 게임을 한서버에서 관리한다고 가정할때
각 게임의 상태와 플레이어를 파악하기 위해
gameId별 게임상황을,
playerId을 이용해서 채팅 로직을 작성

기능

- 입장,퇴장기능에서 생각해볼 거리

방에 입장했을때  내가 방을 생성한경우 , 남이 생성한 방에 들어온 경우 고려

정상퇴장과 비정상퇴장 구분
정상: 나가기 버튼을 누름
비정상: 브라우저 닫기,컴퓨터 종료, 네트워크 끊김

새로고침시 복구메커니즘 작동
뒤로가기시 나가기 모달창 

- 레디기능에서 생각해볼거리
카운터로 레디데이터 저장 레디 +1 , 퇴장 -1 로 데이터를 저장할 경우
-> 로직이 단순해짐




- 채팅기능에서 생각해볼 거리
전송 이벤트 발생-> 모든 클라이언트에게 발신 -> 응답받은 메시지 화면에 렌더링

낙관적 UI 패턴
발신자 화면에는 미리 채팅을 보여준다 -> 응답이 안돌아오면 재전송 활성화 
재전송 메커니즘이 필요


- 게임 흐름에서 구현할 기능

해당턴 좌표 수신 -> 승리판정, 유효성 검사

게임 종료 -> 초기화 트리거 전송

누구턴인지 알려줌
클라이언트로부터 좌표 응답
유효성, 승리판정
누가 이겼는지 알려줌

백엔드 구조에 관한 생각

리액트의 데이터 바인딩을 생각해볼때 흐름은 단방향으로 이루어져야 한다.
그래서 구조를 설계할 때 A <-> B (상호 의존성) 또는 A -> B -> C -> D -> A (순환 의존성) 구조를 피해야 된다.

오류를 전문적으로 처리하는 클래스
RFC 6455의 상태코드 참조

Heartbeat (Ping/Pong) 관리
서버 = 죽은 클라이언트 끊어내서 메모리 확보
클라이언트 = 네트워크상태를 시각적으로 표현
